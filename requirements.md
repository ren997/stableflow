# StableFlow PRD

## 1. 项目概述

### 1.1 项目名称
**StableFlow：面向出海数字商家的 Solana Stablecoin Billing & Reconciliation Infrastructure**

中文可选描述：
**面向出海数字商家的 Solana 稳定币账单、收款与核销基础设施**

### 1.2 项目背景
出海数字商家、SaaS 小团队、自由职业者、跨境服务团队，在全球收款场景中经常遇到以下问题：

- 跨境收款流程复杂
- 传统支付手续费高、到账慢
- 报价、开单、收款、催款、对账分散在多个系统里
- 财务人工核销效率低
- 订阅收费、账单提醒、异常订单处理成本高

稳定币（如 USDC）天然适合全球在线收款场景，但目前大多数商家缺少一个围绕“开单 -> 收款 -> 监听 -> 验证 -> 核销 -> 对账 -> 提醒”的完整工作流系统。

本项目希望构建一个基于 Solana 稳定币支付的账单、支付验证与核销系统，并结合 Agent 运营层能力，帮助商家自动完成账单管理、收款确认、逾期提醒、异常解释和对账总结。

### 1.3 项目目标
构建一个可落地的 MVP 系统，实现以下闭环：

1. 商家创建稳定币账单（Invoice）
2. 系统为每张账单生成唯一收款地址和 Solana 原生支付请求
3. 客户通过 Solana 网络支付 USDC
4. 系统监听链上交易并获取交易详情
5. 系统完成链上支付验证、订单归因与自动核销
6. 系统自动生成通知、支付凭证与基础对账结果
7. 系统提供 Agent 运营层能力辅助创建账单、催款与解释异常

### 1.4 成功标准
满足以下条件时，可认为 MVP 达到可演示、可验证价值的要求：

- 商家可以创建一张 Solana USDC 账单，并获得唯一支付地址与可分享的支付请求
- 客户完成付款后，系统可以识别链上交易、验证支付条件并将账单更新为正确状态
- 商家可以查看已支付、未支付、异常账单及基础汇总数据
- 系统可以对支付成功、过期、异常等关键事件触发通知，并保留可追踪的支付结果
- Agent 可以基于系统已验证的账单与支付数据，完成自然语言开单、异常解释与对账总结

## 2. 用户与场景

### 2.1 核心目标用户
第一阶段重点面向以下用户：

- 出海数字商品商家
- SaaS 小团队
- 自由职业者
- 小型跨境服务团队
- 数字内容创作者

### 2.2 典型用户画像
#### 用户画像 A：SaaS 小团队
- 提供海外订阅服务
- 面向国际客户收费
- 需要按月或按季度开单
- 需要自动提醒续费和对账

#### 用户画像 B：自由职业者 / 外包团队
- 向海外客户提供设计、开发、顾问服务
- 需要按项目或周期收款
- 需要报价、开单、催款、收款确认

#### 用户画像 C：数字商品商家
- 销售模板、插件、会员服务、课程
- 需要低成本全球收款
- 需要简单清晰的财务核销能力

### 2.3 典型使用场景
- 商家为海外客户创建一张 99 USDC 的服务账单，并发送账单链接或支付二维码
- 客户通过 Solana 原生支付流程完成 USDC 支付，系统自动识别、验证并完成核销
- 商家查看本周收款、未支付账单、异常账单和支付证明结果
- 商家通过 Agent 快速生成催款文案，或询问异常订单原因

## 3. 产品定位与边界

### 3.1 产品定位
本产品不是钱包，不是交易所，也不是单纯的聊天机器人。

本产品的定位是：

**一个面向出海数字商家的 Solana Native Stablecoin Billing & Payment Reconciliation Infrastructure。**

Agent 是运营自动化层，核心价值在于：

- 基于 Solana 的稳定币账单能力
- 原生支付请求与支付接入
- 链上支付验证、订单归因与自动核销
- 对账、异常处理与支付结果追踪
- 提高商家运营效率

### 3.2 产品边界
本产品在 MVP 阶段聚焦业务工作流，不提供以下能力：

- 交易所能力
- 法币出金能力
- 自动退款
- 链上智能合约开发
- 多方分账或 Escrow 托管
- 企业级税务系统
- 企业级财务报表
- 高级风控建模

### 3.3 资金与责任边界
为避免产品范围和实现责任不清，MVP 阶段明确以下边界：

- MVP 采用“一单一地址”的订单关联方式，并在此基础上提供可分享的 Solana 原生支付请求
- 平台负责地址分配、链上交易识别、支付验证和账单状态更新
- 平台负责业务层面的收款识别，不承诺提供资产托管、自动退款或法币结算能力
- 地址归集不纳入 MVP，仅作为后续演进方向
- Agent 不参与最终支付判定，支付结果以系统链上验证和核销结果为准

## 5. 核心业务流程

### 5.1 主流程
1. 商家创建 Invoice
2. 系统为该 Invoice 分配唯一收款地址，并生成 Solana 原生支付请求
3. 商家将账单链接、支付请求或二维码发送给客户
4. 客户通过 Solana 钱包使用 USDC 完成支付
5. 系统监听链上交易，并获取交易详情与支付事件
6. 系统校验地址、币种、金额、时间窗口等关键信息，完成订单归因
7. 系统完成自动核销，更新 Invoice 状态并生成支付结果说明
8. 系统通知商家，必要时通知客户
9. 系统将结果纳入对账统计、异常视图和支付证明视图

### 5.2 异常流程
MVP 需要覆盖以下异常情况：

- 少付
- 多付
- 超时付款
- 重复付款
- 非指定币种付款
- 未识别付款
- 到账延迟
- 订单已过期但仍到账

## 6. 核心业务规则

### 6.1 订单与付款关联规则
系统必须能够明确判断“链上这笔付款对应哪个订单”。

MVP 采用：

**每张 Invoice 分配唯一收款地址**

即：
- Invoice A -> Address A
- Invoice B -> Address B

在 MVP 阶段，订单关联以“一单一地址”为主；如支付请求中携带附加标识，则作为辅助验证信息，而不是唯一判断依据。

采用该方案的原因是：

- 实现简单
- 演示清晰
- 不依赖客户填写 memo
- 订单关联最直接
- MVP 风险最低

### 6.2 Invoice 生命周期规则
Invoice 主状态定义如下：

- `DRAFT`：草稿，尚未进入待支付状态
- `PENDING`：已创建并等待支付
- `PAID`：已足额支付且满足有效期要求
- `PARTIALLY_PAID`：已支付但金额不足
- `OVERPAID`：已支付但金额超出应付金额
- `EXPIRED`：账单已过期且未满足正常支付条件
- `FAILED_RECONCILIATION`：核销失败，需进入异常处理

状态规则如下：

- 新创建账单可先进入 `DRAFT`，确认生效后进入 `PENDING`
- 客户足额且在有效期内支付后，账单进入 `PAID`
- 客户少付后，账单进入 `PARTIALLY_PAID`
- 客户多付后，账单进入 `OVERPAID`
- 账单到期且未完成正常支付时，进入 `EXPIRED`
- 核销过程中出现无法完成归因或处理失败的情况时，进入 `FAILED_RECONCILIATION`

### 6.3 异常标签规则
除主状态外，系统需要支持异常标签或异常类型，用于补充说明支付问题。MVP 至少支持以下异常标签：

- `LATE_PAYMENT`：过期后到账
- `WRONG_CURRENCY`：非指定币种支付
- `DUPLICATE_PAYMENT`：重复支付
- `UNMATCHED_PAYMENT`：未识别付款
- `PAYMENT_DELAYED`：到账延迟或确认延迟

说明：

- `LATE_PAYMENT` 不作为主状态，而是异常标签
- 一个账单可以有主状态，同时带有一个或多个异常标签
- 商家后台需要可查看异常标签和异常说明

### 6.4 到期规则
- 每张 Invoice 必须具有明确的到期时间
- 在到期时间之前，账单可正常接收支付并按正常规则核销
- 到期后未完成正常支付的账单进入 `EXPIRED`
- 到期后仍发生支付的，系统需记录交易，并打上 `LATE_PAYMENT` 标签
- 对于过期到账，是否人工确认、是否接受、是否后续补处理，由商家运营策略决定；MVP 先以“标记异常并展示”为主

### 6.5 到账确认与链上验证规则
MVP 对“到账确认”的业务定义如下：

- 系统首先监听到目标地址存在匹配交易
- 系统需要获取该笔交易的链上详情，而不是仅根据地址到账就直接更新状态
- 交易币种为指定币种 USDC
- 交易金额可与账单金额进行比较
- 交易时间可与 Invoice 有效期进行比较
- 交易结果必须可用于判断该交易是否可被本系统采纳为有效支付
- 满足业务校验后，系统进入核销流程并更新状态

说明：

- Agent 不参与到账判定
- 系统必须区分“监听到交易”和“完成链上验证与业务核销”两个阶段
- 对于识别到但暂时无法确认归因的交易，应进入异常视图，而不是直接判定成功
- 监听负责发现交易，验证负责确认交易是否满足该 Invoice 的支付条件

### 6.6 少付、多付、重复支付处理规则
- 若实付金额小于应付金额，账单主状态更新为 `PARTIALLY_PAID`
- 若实付金额大于应付金额，账单主状态更新为 `OVERPAID`
- 若同一账单在已完成一次有效处理后再次收到支付，系统需记录后续交易，并标记 `DUPLICATE_PAYMENT`
- 同一笔链上交易被重复扫描到时，只允许首次成功处理，后续识别为重复处理并跳过

### 6.7 异常单处理原则
以下情况应进入异常单视图：

- 少付
- 多付
- 过期到账
- 非指定币种付款
- 未识别付款
- 核销失败
- 重复支付

异常单处理原则如下：

- 必须保留交易记录、账单状态和异常说明
- 必须允许商家查询和筛选异常单
- MVP 先支持“识别和展示异常”，复杂自动处理不纳入本期范围

### 6.8 幂等规则
以下操作必须保证幂等：

- 支付确认
- 状态更新
- 通知发送
- 异常单处理

当同一笔链上交易多次被扫描到时：

- 仅允许首次成功核销
- 后续处理应识别为重复并跳过
- 系统需要保留处理痕迹，便于排查

## 7. 业务对象定义
本节仅定义业务对象，作为后续技术方案输入，不作为数据库 Schema。

### 7.1 Merchant
表示商家主体。

最小业务信息包括：
- 商家标识
- 商家名称
- 联系邮箱
- 状态

### 7.2 Invoice
表示一张待支付或已支付的账单。

最小业务信息包括：
- 账单编号
- 商家标识
- 客户名称
- 金额
- 币种
- 链
- 到期时间
- 备注或描述
- 主状态
- 异常标签

### 7.3 Payment Address
表示分配给某张 Invoice 的唯一收款地址。

最小业务信息包括：
- 收款地址
- 对应账单
- 指定币种
- 预期金额
- 有效期
- 是否有效

### 7.4 Payment Transaction
表示系统识别到的一笔付款记录。

最小业务信息包括：
- 收款地址
- 付款地址
- 金额
- 币种
- 交易哈希
- 到账时间
- 验证结果
- 识别结果
- 对应账单

### 7.5 Payment Proof
表示一笔支付在系统内的可验证结果视图，用于展示该交易为何被判定为已支付、少付、多付或异常。

最小业务信息包括：
- 对应账单
- 对应交易哈希
- 支付金额与币种
- 支付时间
- 验证结果
- 状态判定结果
- 异常标签或结果说明

### 7.6 Reconciliation Record
表示一次核销处理结果。

最小业务信息包括：
- 对应账单
- 核销结果
- 结果说明
- 处理时间

### 7.7 Notification Record
表示一次通知行为。

最小业务信息包括：
- 对应账单
- 通知类型
- 通知对象
- 发送结果
- 发送时间

## 8. 功能需求

### 8.1 商家模块
功能点：

- 商家注册或登录
- 商家基础信息维护
- 商家查看 Invoice 列表
- 商家查看收款统计
- 商家查看异常账单

输入：
- 商家账户信息
- 商家业务资料

输出：
- 商家主页
- 订单列表
- 收款数据面板
- 异常账单视图

说明：
- MVP 阶段商家账户体系可以简化，但商家身份必须可区分
- 商家只能查看自己的账单、交易和统计结果

### 8.2 Invoice 模块
功能点：

- 创建账单
- 编辑账单
- 查看账单详情
- 设置到期时间
- 设置金额与币种
- 设置客户名称与备注
- 查看当前账单状态与异常标签

说明：
- MVP 默认币种为 USDC
- MVP 默认链为 Solana
- 账单应具备可分享给客户的支付信息

### 8.3 收款地址分配模块
功能点：

- 为每张 Invoice 生成唯一收款地址
- 保存地址与 Invoice 的映射关系
- 管理收款地址状态
- 标记地址是否仍有效

输出：
- paymentAddress
- expectedAmount
- expireAt
- 支付二维码或支付链接所需信息

说明：
- 唯一收款地址是订单归因的主锚点
- 支付请求可在地址基础上附加金额、币种、订单说明等支付信息

### 8.4 支付接入、监听与验证模块
功能点：

- 展示支付地址
- 展示支付金额
- 展示支付二维码或支付链接
- 查询支付状态
- 生成 Solana 原生支付请求
- 监听目标地址到账
- 获取并解析链上交易详情
- 校验币种、金额、时间窗口和订单归因
- 保存交易哈希、到账信息和验证结果
- 回写支付状态

说明：
- 本期不编写链上智能合约
- 本期仅支持 Solana 网络上的 USDC 支付
- 支付接入应同时支持“直接查看支付地址”和“通过支付请求发起支付”
- 监听、验证与业务核销是三个连续但独立的处理阶段

### 8.5 核销模块
功能点：

- 根据链上验证结果确认 Invoice 状态
- 记录支付明细
- 保证幂等更新
- 防止重复核销
- 生成核销结果说明
- 生成可追踪的支付结果或支付证明

说明：
- 核销结果必须可追踪
- 无法自动确认的交易需转入异常单视图

### 8.6 通知模块
功能点：

- 支付成功通知
- 账单过期提醒
- 逾期催款提醒
- 异常订单提醒

通知对象：
- 商家
- 客户（可选）

通知规则：
- 支付成功后，默认通知商家
- 账单到期或异常发生后，默认通知商家
- 客户通知是否发送，由商家配置或操作决定
- 逾期催款支持手动触发；定时自动提醒作为可选增强能力
- Webhook 仅作为后续扩展方向，不属于本期交付能力

### 8.7 对账模块
功能点：

- 查看已支付账单
- 查看未支付账单
- 查看异常账单
- 按日、周、月汇总收款数据
- 按状态筛选账单
- 按异常标签筛选账单

统计口径：
- 默认按支付时间统计收款数据
- 收款总额默认统计成功识别的到账金额
- 多付金额计入收款总额，但需要可单独识别异常金额
- 异常金额至少包含少付、多付、过期到账等异常相关金额
- 过期到账默认纳入异常统计视图，并在汇总中可单独查看

核心输出：
- 总账单数
- 已收款数
- 未支付数
- 异常数
- 收款总额
- 异常金额

### 8.8 Agent 模块
功能定位：

Agent 不负责链上支付本身，负责在已验证的账单与支付结果之上，提供业务操作和运营自动化能力。

MVP 能力：

#### 能力一：自然语言创建账单
示例：
- 给 Alice 开一张 99 USDC 的账单，7 天后到期

要求：
- 能从自然语言中提取客户、金额、币种、到期时间等关键信息
- 输出结构化账单草稿，或直接创建账单结果
- 当信息不完整时，应提示补充，而不是擅自编造关键字段

#### 能力二：自动催款文案生成
示例：
- 生成礼貌提醒
- 生成商务催款
- 生成最后提醒

要求：
- 文案风格应可区分
- 文案内容应基于真实账单信息生成

#### 能力三：异常订单解释
示例：
- 为什么这张账单还没完成支付
- 这周有哪些异常单

要求：
- 解释必须基于系统内已有账单、支付和异常数据
- Agent 可以解释结果，但不负责最终判定支付是否成功

#### 能力四：对账总结
示例：
- 本周收款总额
- 哪些账单已逾期
- 哪些账单存在少付问题

要求：
- 总结必须与对账模块统计口径一致
- 输出应可直接用于商家查看和沟通

## 9. 验收标准

### 9.1 正常流程验收
- 商家可以创建一张 Solana USDC 账单
- 系统可以为该账单生成唯一收款地址和可分享的支付请求
- 客户可以根据支付地址、支付链接或二维码完成付款
- 系统可以识别交易、完成链上验证并将账单更新为 `PAID`
- 商家可以查看交易哈希、支付时间、验证结果和支付结果

### 9.2 异常流程验收
- 对少付账单，系统可识别并更新为 `PARTIALLY_PAID`
- 对多付账单，系统可识别并更新为 `OVERPAID`
- 对过期到账，系统可识别并打上 `LATE_PAYMENT` 标签
- 对重复支付，系统可记录后续交易并标记异常
- 对非指定币种支付，系统可识别并进入异常视图

### 9.3 通知验收
- 支付成功后，系统可向商家发送成功通知
- 账单过期后，系统可向商家发送过期提醒
- 异常订单出现后，系统可向商家发送异常提醒
- 客户通知在启用时可按规则发送

### 9.4 对账验收
- 商家可以查看已支付、未支付、异常账单
- 商家可以查看按日、周、月汇总的收款结果
- 商家可以按状态和异常标签筛选账单
- 对账汇总结果与账单和交易记录保持一致

### 9.5 Agent 验收
- Agent 可根据自然语言创建账单草稿或账单结果
- Agent 可生成至少三种不同风格的催款文案
- Agent 可基于已验证的真实数据解释异常订单原因
- Agent 可输出与对账统计一致的总结结果

## 10. 风险与注意事项

### 10.1 订单与付款关联
必须保证“一单一地址”映射正确，否则会直接影响核销准确性。

### 10.2 金额精度
涉及稳定币支付时必须关注金额精度与展示一致性，避免误判少付或多付。

### 10.3 重复监听
链上事件监听和后续处理必须防止重复核销。

### 10.4 链上验证
系统不能仅凭“某个地址收到钱”就判定支付成功，必须基于交易详情完成验证与归因。

### 10.5 异常支付
少付、多付、过期支付、非指定币种等异常情况必须可识别、可展示、可追踪。

### 10.6 地址归集
后续若收款地址数量增多，需要考虑资金归集方案，但不纳入本期范围。

### 10.7 网络依赖
系统依赖 Solana RPC 节点稳定性，链上确认延迟可能影响状态更新时效。
