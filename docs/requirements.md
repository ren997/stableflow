# StableFlow PRD

## 1. 项目概述

### 1.1 项目名称
**StableFlow：面向出海数字商家的 Solana Stablecoin Billing & Reconciliation Infrastructure**

中文可选描述：
**面向出海数字商家的 Solana 稳定币账单、支付验证与核销基础设施**

### 1.2 项目背景
出海数字商家、SaaS 小团队、自由职业者、跨境服务团队，在全球收款场景中经常遇到以下问题：

- 跨境收款流程复杂
- 传统支付手续费高、到账慢
- 报价、开单、收款、催款、对账分散在多个系统里
- 财务人工核销效率低
- 订阅收费、账单提醒、异常订单处理成本高

稳定币（如 USDC）天然适合全球在线收款场景，但目前大多数商家缺少一个围绕“开单 -> 收款 -> 监听 -> 验证 -> 核销 -> 对账 -> 提醒”的完整工作流系统。

本项目希望构建一个基于 Solana 稳定币支付的账单、支付验证与核销系统，并结合 Agent 运营层能力，帮助商家自动完成账单管理、收款确认、异常解释和基础对账总结。

### 1.3 项目目标
构建一个可落地的 MVP 系统，实现以下闭环：

1. 商家配置固定的 Solana USDC 收款地址
2. 商家创建稳定币账单（Invoice）
3. 系统为每张账单生成唯一 reference 和 Solana 原生支付请求
4. 客户通过 Solana 网络支付 USDC
5. 系统监听链上交易并获取交易详情
6. 系统完成链上支付验证、订单归因与自动核销
7. 系统自动生成支付凭证与基础对账结果
8. 系统提供 Agent 运营层能力辅助创建账单、解释异常和总结结果

### 1.4 成功标准
满足以下条件时，可认为 MVP 达到可演示、可验证价值的要求：

- 商家可以配置固定收款地址，并创建一张 Solana USDC 账单
- 系统可以为该账单生成带唯一 reference 的支付请求
- 客户完成付款后，系统可以识别链上交易、验证支付条件并将账单更新为正确状态
- 商家可以查看已支付、未支付、异常账单及基础汇总数据
- 系统可以保留可追踪的支付结果与 Payment Proof
- Agent 可以基于系统已验证的账单与支付数据，完成自然语言开单、异常解释与对账总结

## 2. 用户与场景

### 2.1 核心目标用户
第一阶段重点面向以下用户：

- 出海数字商品商家
- SaaS 小团队
- 自由职业者
- 小型跨境服务团队
- 数字内容创作者

### 2.2 典型用户画像
#### 用户画像 A：SaaS 小团队
- 提供海外订阅服务
- 面向国际客户收费
- 需要按月或按季度开单
- 需要自动提醒续费和对账

#### 用户画像 B：自由职业者 / 外包团队
- 向海外客户提供设计、开发、顾问服务
- 需要按项目或周期收款
- 需要报价、开单、催款、收款确认

#### 用户画像 C：数字商品商家
- 销售模板、插件、会员服务、课程
- 需要低成本全球收款
- 需要简单清晰的财务核销能力

### 2.3 典型使用场景
- 商家先配置自己的固定 Solana USDC 收款地址
- 商家为海外客户创建一张 99 USDC 的服务账单，并发送账单链接或支付二维码
- 客户通过 Solana 原生支付流程完成 USDC 支付，支付请求中带有唯一 reference
- 系统自动识别、验证并完成核销
- 商家查看本周收款、未支付账单、异常账单和支付证明结果
- 商家通过 Agent 快速生成催款文案，或询问异常订单原因

## 3. 产品定位与边界

### 3.1 产品定位
本产品不是钱包，不是交易所，也不是单纯的聊天机器人。

本产品的定位是：

**一个面向出海数字商家的 Solana Native Stablecoin Billing & Payment Reconciliation Infrastructure。**

Agent 是运营自动化层，核心价值在于：

- 基于 Solana 的稳定币账单能力
- 原生支付请求与支付接入
- 链上支付验证、订单归因与自动核销
- 对账、异常处理与支付结果追踪
- 提高商家运营效率

### 3.2 产品边界
本产品在 MVP 阶段聚焦业务工作流，不提供以下能力：

- 交易所能力
- 法币出金能力
- 自动退款
- 链上智能合约开发
- 多方分账或 Escrow 托管
- 企业级税务系统
- 企业级财务报表
- 高级风控建模

### 3.3 资金与责任边界
为避免产品范围和实现责任不清，MVP 阶段明确以下边界：

- 商家自行配置固定 Solana USDC 收款地址，平台不托管商家私钥
- MVP 采用“单商家固定收款地址 + 每单唯一 reference”的订单关联方式
- 平台负责支付请求生成、链上交易识别、支付验证和账单状态更新
- 未携带 reference 或 reference 不匹配的支付默认进入异常视图
- 地址归集不纳入 MVP，仅作为后续演进方向
- Agent 不参与最终支付判定，支付结果以系统链上验证和核销结果为准

## 5. 核心业务流程

### 5.1 主流程
1. 商家配置固定收款地址
2. 商家创建 Invoice
3. 系统为该 Invoice 生成唯一 reference，并基于商家固定收款地址生成 Solana 原生支付请求
4. 商家将账单链接、支付请求或二维码发送给客户
5. 客户通过 Solana 钱包使用 USDC 完成支付
6. 系统监听链上交易，并获取交易详情与支付事件
7. 系统校验收款地址、reference、币种、金额、时间窗口等关键信息，完成订单归因
8. 系统完成自动核销，更新 Invoice 状态并生成 Payment Proof
9. 系统将结果纳入对账统计、异常视图和支付证明视图
10. Agent 基于已验证结果提供解释或总结

### 5.2 异常流程
MVP 需要覆盖以下异常情况：

- 少付
- 多付
- 超时付款
- 重复付款
- 非指定币种付款
- 未识别付款
- reference 缺失或不匹配
- 到账延迟
- 订单已过期但仍到账

## 6. 核心业务规则

### 6.1 订单与付款关联规则
系统必须能够明确判断“链上这笔付款对应哪个订单”。

MVP 采用：

**单商家固定收款地址 + 每张 Invoice 唯一 reference**

即：
- Merchant A -> Fixed Address A
- Invoice A -> Reference A
- Invoice B -> Reference B

订单归因以“固定地址 + reference”为主；未携带 reference 或 reference 不匹配的支付默认视为异常支付。

采用该方案的原因是：

- 更适合 Solana 原生支付请求
- 降低大量收款地址和私钥管理复杂度
- 更适合黑客松阶段快速做出稳定闭环
- 更容易统一商家资产视图
- 便于后续扩展商家 API、Webhook 和资金归集

### 6.2 Invoice 生命周期规则
Invoice 主状态定义如下：

- `DRAFT`：草稿，尚未进入待支付状态
- `PENDING`：已创建并等待支付
- `PAID`：已足额支付且满足有效期要求
- `PARTIALLY_PAID`：已支付但金额不足
- `OVERPAID`：已支付但金额超出应付金额
- `EXPIRED`：账单已过期且未满足正常支付条件
- `FAILED_RECONCILIATION`：核销失败，需进入异常处理

状态规则如下：

- 新创建账单可先进入 `DRAFT`，确认生效后进入 `PENDING`
- 客户足额且在有效期内支付后，账单进入 `PAID`
- 客户少付后，账单进入 `PARTIALLY_PAID`
- 客户多付后，账单进入 `OVERPAID`
- 账单到期且未完成正常支付时，进入 `EXPIRED`
- 核销过程中出现无法完成归因或处理失败的情况时，进入 `FAILED_RECONCILIATION`

### 6.3 异常标签规则
除主状态外，系统需要支持异常标签或异常类型，用于补充说明支付问题。MVP 至少支持以下异常标签：

- `LATE_PAYMENT`：过期后到账
- `WRONG_CURRENCY`：非指定币种支付
- `DUPLICATE_PAYMENT`：重复支付
- `UNMATCHED_PAYMENT`：未识别付款
- `MISSING_REFERENCE`：未携带 reference
- `INVALID_REFERENCE`：reference 不匹配
- `PAYMENT_DELAYED`：到账延迟或确认延迟

说明：

- `LATE_PAYMENT` 不作为主状态，而是异常标签
- 一个账单可以有主状态，同时带有一个或多个异常标签
- 商家后台需要可查看异常标签和异常说明

### 6.4 到期规则
- 每张 Invoice 必须具有明确的到期时间
- 在到期时间之前，账单可正常接收支付并按正常规则核销
- 到期后未完成正常支付的账单进入 `EXPIRED`
- 到期后仍发生支付的，系统需记录交易，并打上 `LATE_PAYMENT` 标签
- 对于过期到账，是否人工确认、是否接受、是否后续补处理，由商家运营策略决定；MVP 先以“标记异常并展示”为主

### 6.5 到账确认与链上验证规则
MVP 对“到账确认”的业务定义如下：

- 系统首先监听到商家固定收款地址上的候选交易，或基于 reference 查询候选交易
- 系统需要获取该笔交易的链上详情，而不是仅根据地址到账就直接更新状态
- 交易需要能够匹配到该 Invoice 对应的 reference
- 交易币种为指定币种 USDC
- 交易金额可与账单金额进行比较
- 交易时间可与 Invoice 有效期进行比较
- 交易结果必须可用于判断该交易是否可被本系统采纳为有效支付
- 满足业务校验后，系统进入核销流程并更新状态

说明：

- Agent 不参与到账判定
- 系统必须区分“监听到交易”和“完成链上验证与业务核销”两个阶段
- 对于缺少 reference 或 reference 不匹配的交易，应进入异常视图，而不是直接判定成功
- 监听负责发现交易，验证负责确认交易是否满足该 Invoice 的支付条件

### 6.6 少付、多付、重复支付处理规则
- 若实付金额小于应付金额，账单主状态更新为 `PARTIALLY_PAID`
- 若实付金额大于应付金额，账单主状态更新为 `OVERPAID`
- 若同一账单在已完成一次有效处理后再次收到支付，系统需记录后续交易，并标记 `DUPLICATE_PAYMENT`
- 同一笔链上交易被重复扫描到时，只允许首次成功处理，后续识别为重复处理并跳过

### 6.7 异常单处理原则
以下情况应进入异常单视图：

- 少付
- 多付
- 过期到账
- 非指定币种付款
- 未识别付款
- 缺少 reference
- reference 不匹配
- 核销失败
- 重复支付

异常单处理原则如下：

- 必须保留交易记录、账单状态和异常说明
- 必须允许商家查询和筛选异常单
- MVP 先支持“识别和展示异常”，复杂自动处理不纳入本期范围

### 6.8 幂等规则
以下操作必须保证幂等：

- 支付确认
- 状态更新
- 异常单处理
- Payment Proof 生成

当同一笔链上交易多次被扫描到时：

- 仅允许首次成功核销
- 后续处理应识别为重复并跳过
- 系统需要保留处理痕迹，便于排查

## 7. 业务对象定义
本节仅定义业务对象，作为后续技术方案输入，不作为数据库 Schema。

### 7.1 Merchant
表示商家主体。

最小业务信息包括：
- 商家标识
- 商家名称
- 联系邮箱
- 固定收款地址配置
- 状态

### 7.2 Invoice
表示一张待支付或已支付的账单。

最小业务信息包括：
- 账单编号
- 公共访问标识
- 商家标识
- 客户名称
- 金额
- 币种
- 链
- 到期时间
- 备注或描述
- 主状态
- 异常标签

### 7.3 Payment Request
表示分配给某张 Invoice 的支付请求。

最小业务信息包括：
- 固定收款地址
- 对应账单
- unique reference
- 指定币种
- 预期金额
- 有效期
- 支付链接

### 7.4 Payment Transaction
表示系统识别到的一笔付款记录。

最小业务信息包括：
- 收款地址
- 付款地址
- reference
- 金额
- 币种
- 交易哈希
- 到账时间
- 验证结果
- 识别结果
- 对应账单

### 7.5 Payment Proof
表示一笔支付在系统内的可验证结果视图，用于展示该交易为何被判定为已支付、少付、多付或异常。

最小业务信息包括：
- 对应账单
- 对应交易哈希
- 支付金额与币种
- 支付时间
- 验证结果
- 状态判定结果
- 异常标签或结果说明

### 7.6 Reconciliation Record
表示一次核销处理结果。

最小业务信息包括：
- 对应账单
- 核销结果
- 结果说明
- 处理时间

## 8. 功能需求

### 8.1 商家模块
功能点：

- 商家注册或登录
- 商家基础信息维护
- 商家固定收款地址配置
- 商家查看 Invoice 列表
- 商家查看收款统计
- 商家查看异常账单

说明：
- MVP 阶段商家账户体系可以简化，但商家身份必须可区分
- 商家只能查看自己的账单、交易和统计结果

### 8.2 Invoice 模块
功能点：

- 创建账单
- 编辑账单
- 查看账单详情
- 设置到期时间
- 设置金额与币种
- 设置客户名称与备注
- 查看当前账单状态与异常标签

说明：
- MVP 默认币种为 USDC
- MVP 默认链为 Solana
- 账单应具备可分享给客户的支付信息和唯一 reference

### 8.3 支付请求模块
功能点：

- 基于商家固定收款地址生成支付请求
- 为每张 Invoice 生成唯一 reference
- 保存支付请求与 Invoice 的映射关系
- 生成支付二维码或支付链接

输出：
- fixedPaymentAddress
- reference
- expectedAmount
- expireAt
- 支付二维码或支付链接所需信息

说明：
- 固定收款地址是商家资产入口
- reference 是订单归因主锚点

### 8.4 支付接入、监听与验证模块
功能点：

- 展示支付地址
- 展示支付金额
- 展示支付二维码或支付链接
- 查询支付状态
- 生成 Solana 原生支付请求
- 监听目标地址到账
- 获取并解析链上交易详情
- 校验 reference、币种、金额、时间窗口和订单归因
- 保存交易哈希、到账信息和验证结果
- 回写支付状态

说明：
- 本期不编写链上智能合约
- 本期仅支持 Solana 网络上的 USDC 支付
- 支付接入应同时支持“直接查看支付信息”和“通过支付请求发起支付”
- 监听、验证与业务核销是三个连续但独立的处理阶段

### 8.5 核销模块
功能点：

- 根据链上验证结果确认 Invoice 状态
- 记录支付明细
- 保证幂等更新
- 防止重复核销
- 生成核销结果说明
- 生成可追踪的 Payment Proof

说明：
- 核销结果必须可追踪
- 无法自动确认的交易需转入异常单视图

### 8.6 对账模块
功能点：

- 查看已支付账单
- 查看未支付账单
- 查看异常账单
- 按日、周、月汇总收款数据
- 按状态筛选账单
- 按异常标签筛选账单

统计口径：
- 默认按支付时间统计收款数据
- 收款总额默认统计成功识别的到账金额
- 多付金额计入收款总额，但需要可单独识别异常金额
- 异常金额至少包含少付、多付、过期到账、未识别支付等异常相关金额
- 过期到账默认纳入异常统计视图，并在汇总中可单独查看

### 8.7 Agent 模块
功能定位：

Agent 不负责链上支付本身，负责在已验证的账单与支付结果之上，提供业务操作和运营自动化能力。

MVP 能力：

- 自然语言创建账单
- 异常订单解释
- 对账总结

说明：
- 催款文案和复杂通知策略不作为 P0 必需项
- Agent 输出必须基于已验证的真实数据

## 9. 验收标准

### 9.1 正常流程验收
- 商家可以配置一条固定收款地址
- 商家可以创建一张 Solana USDC 账单
- 系统可以为该账单生成唯一 reference 和可分享的支付请求
- 客户可以根据支付链接或二维码完成付款
- 系统可以识别交易、完成链上验证并将账单更新为 `PAID`
- 商家可以查看交易哈希、验证结果和 Payment Proof

### 9.2 异常流程验收
- 对少付账单，系统可识别并更新为 `PARTIALLY_PAID`
- 对多付账单，系统可识别并更新为 `OVERPAID`
- 对过期到账，系统可识别并打上 `LATE_PAYMENT` 标签
- 对重复支付，系统可记录后续交易并标记异常
- 对缺少 reference 或 reference 错误的支付，系统可识别并进入异常视图

### 9.3 对账验收
- 商家可以查看已支付、未支付、异常账单
- 商家可以查看按日、周、月汇总的收款结果
- 商家可以按状态和异常标签筛选账单
- 对账汇总结果与账单和交易记录保持一致

### 9.4 Agent 验收
- Agent 可根据自然语言创建账单草稿或账单结果
- Agent 可基于已验证的真实数据解释异常订单原因
- Agent 可输出与对账统计一致的总结结果

## 10. 风险与注意事项

### 10.1 订单与付款关联
必须保证“固定收款地址 + reference”映射正确，否则会直接影响核销准确性。

### 10.2 金额精度
涉及稳定币支付时必须关注金额精度与展示一致性，避免误判少付或多付。

### 10.3 重复监听
链上事件监听和后续处理必须防止重复核销。

### 10.4 链上验证
系统不能仅凭“某个地址收到钱”就判定支付成功，必须基于交易详情和 reference 完成验证与归因。

### 10.5 支付路径依赖
若客户绕过支付请求手动转账且未携带 reference，系统可能无法自动归因，应进入异常视图。

### 10.6 网络依赖
系统依赖 Solana RPC 节点稳定性，链上确认延迟可能影响状态更新时效。
